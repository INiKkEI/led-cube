# ESP32 Pin Budget

## Purpose
This document defines the expected GPIO usage for Version 1 of the LED cube controller. Its goal is to confirm that the selected ESP32 solution has enough usable pins for display control, programming, debugging, and the single user input button before schematic capture begins.

## Design assumptions
This pin budget is based on the following architectural decisions:

- ESP32 is the main controller
- 64 column outputs are generated by a serial driver chain
- layer MOSFET control is also provided through the driver chain
- one physical button is used to switch between animations
- serial debug is included
- firmware is loaded through the normal ESP32 programming interface
- no advanced external communication interface is required for Version 1

## Required signal groups

### 1. Driver-chain control signals
The ESP32 must control the serial driver chain that generates both column outputs and layer-select outputs.

Minimum expected signals:

- serial data output
- shift clock
- latch / storage clock
- output enable / blanking control

Recommended count:

`4 GPIO`

### Why these are needed
- **data** shifts the output pattern into the chain
- **shift clock** advances bits through the chain
- **latch** transfers shifted data to the active outputs
- **output enable / blank** is strongly recommended to suppress ghosting during updates and layer switching

Although the design might work without a dedicated blanking signal in some architectures, reserving it now is the safer choice.

## 2. Serial debug / programming
Version 1 includes serial debug. The ESP32 also needs a programming path.

Minimum expected signals:

- UART TX
- UART RX

Recommended count:

`2 GPIO`

### Why these are needed
- firmware upload and serial logging are much easier with a normal UART path
- bring-up is faster and safer when serial debug is available from the first revision

## 3. User input
Version 1 includes one physical button for switching between preloaded animations.

Required signals:

- one digital input for the animation-select button

Recommended count:

`1 GPIO`

### Notes
This input should have a defined idle state using an internal or external pull-up or pull-down resistor. Firmware should debounce the button.

## 4. Optional but useful reserved pins
Even if not strictly required in the first schematic draft, it is good practice to leave a small amount of margin for bring-up or future adjustment.

Recommended reserve:

- 1 spare GPIO for test, status, or future expansion

Recommended count:

`1 GPIO`

## Total required GPIO count

### Core required pins
- driver chain control: 4
- UART debug/programming: 2
- animation button: 1

`Core total = 7 GPIO`

### With one spare pin reserved
- core required: 7
- spare reserved: 1

`Planned total = 8 GPIO`

## What is not required because of the chosen architecture
The following are **not** required as direct ESP32 GPIOs:

- 64 direct column control outputs
- 8 direct layer MOSFET gate control outputs

This is the main reason the design is feasible with an ESP32. The serial driver chain greatly reduces the required GPIO count.

## Pin-function summary table

| Function group | Signals | GPIO count |
|---|---|---:|
| Driver-chain control | Data, shift clock, latch, output enable | 4 |
| UART debug/programming | TX, RX | 2 |
| User input | Animation button | 1 |
| Reserved margin | Spare GPIO | 1 |
| **Total planned** |  | **8** |

## Practical allocation rules
At this stage, the goal is not to freeze exact GPIO numbers yet, but to define the constraints for later assignment.

### Use suitable pins for high-speed control
The driver-chain signals should be assigned to GPIOs that are safe for regular digital output and suitable for stable timing.

### Protect programming and boot behavior
Pins involved in boot mode selection or other startup constraints should not be assigned casually to signals that can force an invalid startup state.

### Keep UART accessible
UART pins should remain easy to reach in the schematic and PCB so flashing and debugging are simple during bring-up.

### Keep the button on a simple input-capable pin
The animation button should use a GPIO that behaves predictably at reset and does not interfere with boot.

## Boot and constraint notes
Exact ESP32 pin selection depends on the specific ESP32 board or module used. Before freezing the schematic, check the selected ESP32 documentation for:

- boot strapping pins
- pins that must not be pulled the wrong way at reset
- pins reserved by onboard flash or other hardware
- limits of any onboard USB-UART or programming interface
- which pins are safely exposed on the selected board or module carrier

Because of this, the architecture document should freeze the **pin count and function groups** first, and the exact GPIO numbers second.

## Architecture consequences
This pin budget confirms that the chosen control method is practical.

Because both the column outputs and layer-select control are produced through the serial driver chain, the ESP32 only needs a small number of GPIOs for the display interface. This leaves enough room for:

- serial debug
- one button input
- at least a small expansion margin

This is a strong argument in favor of the selected architecture for Version 1.

## Frozen Phase 2 conclusion
For Version 1, the controller should reserve:

- 4 GPIO for driver-chain control
- 2 GPIO for UART debug/programming
- 1 GPIO for the animation-select button
- 1 additional GPIO as margin

So the design target is:

`8 planned GPIO`

This is well within the capability of a practical ESP32 solution, provided the final GPIO assignment respects boot and board-level pin constraints.

## What still needs to be decided later
This document does not yet freeze:

- exact ESP32 module or board variant
- exact GPIO numbers
- whether driver-chain control is implemented by bit-banging or hardware SPI
- whether UART flashing/debug uses a dedicated USB-UART interface on the controller PCB or an external header
- whether the spare GPIO remains unused or is assigned later

These details belong to component selection and schematic capture, not to the pin-budget phase.
